# Benchmark: Kimi K2.5 3-bit (1T params, 32B active)

## Hardware
- Apple M5 Max, 128GB unified memory
- macOS Sequoia 16.4
- SSD: ~36 GB/s sequential read

## Model
- Kimi K2.5 (KimiK25ForConditionalGeneration)
- DeepSeek V3 derivative architecture
- Total params: ~1T, Active per token: 32B
- 61 layers, 256 routed experts + 1 shared, top-8
- 91 safetensors shards, 418GB at 3-bit quantization
- mlx-lm has native model support (`mlx_lm.models.kimi_k25`)

## Results

### Approach 1: mlx-lm standard load (mmap)

**Result: FAILED — Metal GPU Timeout**

```
libc++abi: terminating due to uncaught exception of type std::runtime_error:
[METAL] Command buffer execution failed: Caused GPU Timeout Error
(00000002:kIOGPUCommandBufferCallbackErrorTimeout)
```

The model loaded successfully into memory-mapped safetensors (lazy=True),
but the first forward pass triggered a GPU timeout. The 418GB model on
128GB RAM requires ~70% of weights to be paged from SSD on every forward
pass. The page fault latency during Metal command buffer execution exceeds
the GPU timeout deadline.

This confirms that **standard mlx-lm mmap is not viable for models >~2x RAM**.
The Qwen 397B baseline (209GB, ~1.6x RAM) works because the page cache hit
rate is high enough. At 3.3x RAM, Kimi K2.5 crosses the threshold.

### Approach 2: flash-moe SSD streaming (not yet attempted)

Requires porting flash-moe's Metal+pread architecture to Kimi K2.5.
Estimated 2-3 days. Key challenge: Kimi uses MLA attention (latent
attention with kv_lora_rank=512) vs standard GQA in Qwen.

### Approach 3: llama.cpp partial offload (not yet attempted)

Requires GGUF conversion of Kimi K2.5 weights. Status unknown.

## Conclusion

Kimi K2.5 3-bit cannot run on M5 Max 128GB via mlx-lm's standard path.
SSD streaming (flash-moe port or equivalent) is required to hit the 8 tok/s
target. This validates the sylveste-14g epic's premise that a generalized
SSD streaming engine is needed.
